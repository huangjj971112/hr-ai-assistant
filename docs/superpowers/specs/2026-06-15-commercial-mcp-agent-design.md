# Commercial MCP Agent Design

## Goal

Upgrade the HR MCP proof of concept into a production-oriented agent path with
strict model-driven tool selection, versioned and idempotent pending leave
confirmation, unified tool audit logs, trace IDs, validation, and permission
tests.

## Strict Model-Driven Agent

`POST /api/ai/mcp/chat` must use a configured `ChatModel`. The agent obtains MCP
tool callbacks from the HR MCP Server and lets the model choose tools. It never
falls back to keyword routing or the local Agent.

Security-owned fields are injected after model selection:

- `toolToken`
- `sessionId`
- `traceId`
- `idempotencyKey`

When no model is configured, return `MCP_MODEL_UNAVAILABLE`. Model execution
failures return `MCP_MODEL_CALL_FAILED`.

## Pending Leave Contract

Pending leave state remains in Redis through `AgentMemoryService`, and adds:

- `pendingId`: UUID generated by the server
- `version`: starts at `1`
- `status`: `PENDING_CONFIRMATION`, `SUBMITTED`, or `CANCELLED`
- submitted response and the successful idempotency key

Confirmation requires `pendingId`, `expectedVersion`, and `idempotencyKey`.
The pending record is bound to tenant, user, and session. Repeating the same
idempotency key returns the original submitted response. A stale version returns
`PENDING_VERSION_CONFLICT`.

## Audit And Trace

Every MCP tool invocation gets a `traceId`, returned in `ToolResult` and stored
in `mcp_tool_audit_log`. Audit fields include tool name, tenant, user, session,
pending ID, status, error code, duration, and a sanitized argument summary.
Tokens and full leave reasons are never persisted.

## Validation And Errors

All tools validate required request objects and fields before calling business
services. Stable errors include:

- `VALIDATION_FAILED`
- `INVALID_TOOL_TOKEN`
- `TOOL_TOKEN_EXPIRED`
- `FORBIDDEN`
- `PENDING_NOT_FOUND`
- `PENDING_VERSION_CONFLICT`
- `IDEMPOTENCY_CONFLICT`
- `MCP_MODEL_UNAVAILABLE`
- `MCP_MODEL_CALL_FAILED`
- `MCP_TOOL_CALL_FAILED`

## Compatibility

The six MCP tool names and Redis Memory mechanism remain. The frontend pending
card remains compatible, with added pending ID and version fields used by the
backend agent during confirmation.
